**Reviewer**: meta-prism (Quality Forensics)

**Target tag**: v2.2.0 (commit a3bb99b0)
**Scope**: design doc + contract + 4 PoC modules + 4 test files + progress + CHANGELOG

---

## 1. Verdict

**PASS-WITH-FINDINGS**

v2.2.0 achieves its stated scope (Phase 1 design layer + isolated PoC). All four user-locked decisions (Q1-Q4) have concrete implementations with passing tests. The five ironclad rules are honored at the contract/PoC layer. The release carries real drift between (a) the new contract vocabulary and the production hooks the design plans to eventually replace, and (b) several rule semantics whose tests cover the happy path but not boundary conditions. None of the findings block the tag, but several must be fixed before R4 (v2.3.0) wires PoC modules into the runtime, otherwise the hardcoding will reappear under new names.

Caveat: the aggregator order `{pass:0, skip:1, warn:2, block:3}` at `:95` is tested for block-dominates, all-pass, warn-dominates-pass+skip; the boundary `skip dominates pass` (when only `not_applicable_with_reason` rules exist and all skip) is not directly asserted.

### Q3: v1.0 supports zh / en / ja / ko (aligned with README)

## 4. Drift Detection

### D1: Rule-ID vs Packet-Name vocabulary mismatch (HIGH)

The contract uses rule IDs like `testStrategyDefined`, `rollbackPlanDefined`, `structureHygiene`, `interfaceContract`, `sideEffectLedger`, `permissionMatrix` (`deliverable-type-profiles.json:82-89, 107-111, 130-135`). The production hook uses *packet* names with a `Packet` suffix: `testStrategyPacket`, `structureHygienePacket`, `permissionMatrixPacket`, `sideEffectLedgerPacket`, `rollbackPlanPacket` (`spine-state.mjs:433-447`). When R4 wires the registry into the hook, there must be a mapper, otherwise rule IDs and packet names will never join. Neither the design doc nor any v2.2.0 code specifies that mapping. This is a real future-coupling defect; not yet harmful in v2.2.0 but guaranteed to bite in v2.3.0.

### D2: not_applicable_with_reason semantics already in production (LOW)

`spine-state.mjs:447` `rollbackPlanPacket: [ready, not_applicable_with_reason]` already accepts the string `not_applicable_with_reason` as a status value. So the 4-tier severity model is partly pre-existing as a packet status vocabulary item, not new. The design doc treats Q2 as the binary model is rejected, but the actual upgrade is from binary + one status escape valve to first-class 4-tier severity with explicit decision semantics. Worth restating in design section 3.4 for accuracy.

### D3: Hook still uses English-only regex (KNOWN, ACCEPTED)

`enforce-agent-dispatch.mjs:240` regex unchanged; the lexicon module is not yet imported. Acknowledged by design doc R2 and CHANGELOG. No action required for v2.2.0; must move in R2 (v2.3.0).

### D4: Inference thresholds vs contract thresholds (LOW)

Contract `deliverable-type-profiles.json:33` declares `confidenceThresholds: { high: 0.85, medium: 0.6, low: 0.0 }` as decimals from 0 to 1. The inference function `deliverable-type-profile.mjs:130-133` uses integer score thresholds `top.score >= 4 && margin >= 2` for high; `top.score >= 2 && margin >= 1` for medium. These two scales are not linked. Either rename the contract field or reshape the inference function to honor the declared thresholds. Right now the contract field is decoration.

### D5: intentLexicon.version declared but only weakly enforced (LOW)

Contract has `intentLexicon.version: 1.0.0` at `:39`. Lexicon module loads it at `intent-verb-lexicon.mjs:47` as `intentLexiconRaw.version ?? null`. There is no semver compatibility check. If a future contract bumps to `2.0.0` with breaking word-list structure changes, the loader will silently accept it.

## 5. Anti-Slop Audit

### S1: SLOP-05 unsourced conclusion (LOW)

Progress doc states `npm run meta:check 20/20 pass`. The 20/20 figure is not cross-checked anywhere in the artifacts; the design doc references `npm run meta:verify:all` and `npm run meta:check` without saying how many checks each runs. Low severity; easily verified by re-running, but presented as evidence without citation. Recommendation: capture the actual output and inline it in `tests/poc-design-gate/RESULTS.md`.

### S2: Empty-concept risk on R4-R8 paths (LOW)

Design doc `:212-220` lists R4-R8 with one-line descriptions and feature-flag env var names (e.g. `META_KIM_GATE_DISPATCHER=1`). None have acceptance tests or schemas. Honest forward-pointers, not deliverables; acceptable but should not be mistaken for a contract.

### S3: Test happy-path skew (MEDIUM)

48 tests is strong for a Phase 1 release. The skew:

- Boundary cases tested: null deliverable type, missing config file, duplicate registration, frozen-registry-register, whitespace skipReason, empty profiles, empty text.
- Boundary cases NOT tested: malformed JSON file (parse_error path at `deliverable-type-profile.mjs:34` unreached); profile entries with missing `type` field (`:46` unreached); profiles whose `rules` arrays themselves contain malformed entries; lexicon entries with non-string word lists; `dispatchProfileGates` when only `skip` decisions are present.

Each unreached error path is a missed assertion. None are P0, but `RESULTS.md:20` advertises covering all four user decisions plus error paths, which overclaims.

### S4: Replaceability (PRIN-04 / SLOP-06)

Mental rename test: would the design read coherently if `DeliverableTypeProfile` were globally renamed to `WorkItemKind`? Yes; the abstractions are domain-generic. Acceptable for a contract module. No SLOP-06 flag because the design is genuinely about a *domain* (governance of typed deliverables), not a concrete task. The abstractions are reusable, not domain-bound; correct for a framework.

### S5: Naming consistency (LOW)

- Severity level names mix kebab-case with snake-case: `required-strict` vs `not_applicable_with_reason`. Pick one (`required_strict` or `not-applicable-with-reason`); do not mix.
- Decision names (pass/warn/block/skip) consistent.
- File names kebab-case `.mjs` consistent.
- JSON severity values hyphenated; JSON keys camelCase; acceptable JS convention.

The mixed `required-strict` + `not_applicable_with_reason` is a small but real inconsistency; fix before R4 freezes naming.

### S6: SLOP-03 empty-concept check

Not detected. Every abstraction has a stated implementer module + at least one test. No bare promises.

### S7: SLOP-07 fabricated-data check

Traced every numeric claim:
- 48 tests, 0 fail: `RESULTS.md:7-9`; matches per-file totals (9+11+14+14 = 48).
- 16 word lists: `intentLexicon.intents` has 4 intents x 4 languages = 16 nested arrays at `deliverable-type-profiles.json:40-65`.
- 5 standard profiles: counted in JSON; asserted by `01-deliverable-type-profile.test.mjs:21-25`.

No fabrication detected.

## 6. Findings (severity-ranked)

### CRITICAL: none

The release does not block tag retention. v2.2.0 stands.

### HIGH

- **H1: Rule-ID vs packet-name vocabulary unmapped**. Before R4 wires PoC modules into the production hook, ship an explicit `ruleId -> packetName` mapper either as a third config block in `deliverable-type-profiles.json` or as a helper in `policy-registry.mjs`. Without it, R4 will reintroduce hardcoding to bridge the two namespaces.
- **H2: Q4 first-write confirmation is data-only, not enforced**. `requiresConfirmation: true` is computed but no PreToolUse hook reads it. Either land a stub interceptor in v2.3.0 first commit, or update `progress-v2.2.0.md:23` phrasing to make clear that Q4 enforcement defers to R7.
- **H3: Inference threshold magic numbers not driven by contract**. `deliverable-type-profile.mjs:130-133` ignores the declared `confidenceThresholds` in the contract. Either remove the field from the contract or refactor the function to consume it. Today the contract is misleading.

### LOW

- **L1**: Severity naming inconsistency; pick one of `required-strict` / `required_strict` / `requiredStrict` and apply uniformly.
- **L2**: Test suite advertises error paths but skips `parse_error`, `schema_error`, lexicon malformed entries. Add 5-8 negative-case tests.
- **L3**: `meta:check 20/20` claim not citation-anchored; capture command output in `RESULTS.md`.
- **L4**: `intentLexicon.version` semver not enforced; add a compatibility gate in `loadLexicon`.
- **L5**: Aggregator skip-dominates-pass not directly tested in `03-gate-dispatcher.test.mjs`; add one case.
- **L6**: `resolveProfile` accepts `undefined` / empty string / non-string but only `null` is tested; add 2 cases.
- **L7**: Design doc section 3.4 frames Q2 as binary rejected but `spine-state.mjs:447` already had `not_applicable_with_reason` as a status string. Restate as upgrade to first-class 4-tier model.

## 7. v2.3.0 Recommendations (priority-ordered)

1. **Land the rule-ID to packet-name mapper before any R4 work**. Add `ruleToPacketMap` block in `deliverable-type-profiles.json` and consume from `gate-dispatcher.mjs`. Blocks all later wiring.
2. **R2 first, R6 second**. The intent lexicon migration (`enforce-agent-dispatch.mjs:240` regex -> `loadLexicon`) is the highest-visibility user-facing change (zh/en/ja/ko users immediately benefit) and the smallest blast radius. R6 (unknown-type clarification flow) is the second smallest because it only triggers on the unknown path.
3. **Make Q4 enforcement real**. Stub PreToolUse handler that reads `requiresConfirmation` and, if true, requires `deliverableType` confirmation in spine state before allowing `Edit`, `Write`, or `MultiEdit`.
4. **Reconcile threshold scales**. Either drop `confidenceThresholds` from contract or refactor inference to honor it. Do not ship R7 with two parallel scoring systems.
5. **Add negative-path test coverage to tests/poc-design-gate/**. Target a 60+ test suite by v2.3.0 with the boundary cases listed in L2/L5/L6.
6. **Lock severity naming**. Pick one convention; touch this only once.
7. **Open an evolution writeback for the _qN_decision_evidence pattern**. The contract inline annotations (`_q1_decision_evidence` ... `_q4_decision_evidence`) are a clever cross-reference; document the convention so future profiles maintain it.
