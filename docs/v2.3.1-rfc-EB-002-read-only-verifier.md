# RFC: EB-002 — `read_only_verifier` Capability Slot

**Status**: draft
**Target release**: v2.3.1
**Author**: main-thread (post-v2.3.0 carry-forward)
**Created**: 2026-05-26
**Related bug surface**: `canonical/runtime-assets/shared/hooks/spine-state.mjs` (read_only_verifier slot + dispatchChain auto-append) + `enforce-agent-dispatch.mjs` (gate insertion)

## 1. Problem statement

Reviewer subagents dispatched in `review` / `meta_review` / `verification` stages cannot independently re-run verify commands (`git diff`, `git log --stat`, `npm run meta:check`, `node --check`, `node -e 'JSON.parse(...)'`). The `enforce-agent-dispatch.mjs` PreToolUse hook denies all Bash invocations in these stages because:

1. `checkStageRequirements(state)` returns `met: false` when `dispatchChain.<stage>` does not yet include the required meta-agent name.
2. The dispatcher infrastructure does not auto-append `ownerAgent` to `dispatchChain.<stage>` upon Agent tool dispatch — main thread must hand-edit spine state.
3. The `recordDispatch` function in `spine-state.mjs` does not auto-append the `ownerAgent` of a matched `workerTaskPackets[]` packet to `dispatchChain[currentStage]` when dispatching to a non-governance subagent (e.g., `general-purpose`). This forces hand-edits to populate the audit trail and is tracked as HOOK-INFRA-001.
4. Even with `dispatchChain` populated, the readonly bash whitelist (`bash-readonly-whitelist.mjs`) is checked *after* the stage-requirements gate, so whitelisted commands still fall through to deny.

**First-hand evidence from v2.3.0**: W2 (meta-prism review) and W3 (meta-warden meta-review + verification) both reported `successMarkerFormat: "exit-code-only", status: "skipped"` with `skipReason` citing the hook deny. W3 substituted file-level `Read` + `Glob` verification but could not produce independent stdout-text evidence for `npm run meta:check`. This forced reliance on main-thread's pre-dispatch stdout capture, weakening the Rule 16/17 dogfood chain.

## 2. Goals

- Allow reviewer/arbiter agents to run a bounded set of read-only verify commands in `review` / `meta_review` / `verification` stages without main-thread intervention.
- Keep all mutating commands (`git push`, `git commit`, `npm install`, file writes via Bash) denied in these stages.
- Preserve the capability-first / dispatch-chain governance for non-verify execution work.
- Provide a single source of truth for "what a read-only verifier may run" so the policy doesn't drift across hook updates.

## 3. Non-goals

- Do NOT replace the `bash-readonly-whitelist.mjs` general read-only allowance for `critical` / `fetch` / `thinking` stages — those already work.
- Do NOT broaden write/mutate permissions in any stage.

> **Note**: HOOK-INFRA-001 (dispatchChain.<stage> auto-append on Agent dispatch) is consolidated into this RFC v2 because it shares the same fix surface (`spine-state.mjs`) and unblocks the same governance gap. See Section 4.3.

## 4. Proposed design

### 4.1 New capability slot in `STAGE_META_AGENT_MAP`

`canonical/runtime-assets/shared/hooks/spine-state.mjs` extends `STAGE_META_AGENT_MAP[stage]` entries with a new field:

```javascript
review: {
  required: ["meta-prism"],
  label: "Review (Prism quality forensics)",
  readOnlyVerifierEnabled: true,           // NEW
  readOnlyVerifierCommands: [               // NEW (or inherit from default)
    "git diff",
    "git diff --stat",
    "git log",
    "git show",
    "git status",
    "npm run meta:check",
    "npm run meta:test:meta-theory",
    "node --check",
    "node -e",
    "node --test"
  ]
}
```

Default behavior when `readOnlyVerifierEnabled` is unset: deny (current behavior).

### 4.2 New gate in `enforce-agent-dispatch.mjs`

Insert between current capability-first gate and stage-requirements gate:

```javascript
// read_only_verifier gate
if (toolName === "Bash" && !state.queryBypass && !state.simpleMode) {
  const stage = state.currentStage;
  const stageConfig = STAGE_META_AGENT_MAP[stage];
  const cmd = toolInput?.command || "";

  if (stageConfig?.readOnlyVerifierEnabled) {
    const whitelist = stageConfig.readOnlyVerifierCommands || [];
    const allowed = whitelist.some((prefix) => cmd.startsWith(prefix));
    if (allowed) {
      // Pass through immediately; do not run stage-requirements check
      process.exit(0);
    }
  }
}
```

### 4.3 HOOK-INFRA-001 — Auto-append ownerAgent to dispatchChain on Agent dispatch

**Current bug**: `recordDispatch(state, agentDesc, metaName)` appends `metaName` to `dispatchChain.<currentStage>[]` only when text scanning extracted a governance agent name. The hand-edit gap surfaces when:
- Dispatch description does not literally contain a governance agent string
- Dispatch is to a non-governance subagent (e.g., `general-purpose`) whose `workerTaskPackets[].ownerAgent` is the authoritative record

**Evidence from v2.3.0**: Reviewer dispatches were auto-appended when description used the governance agent's name. Subsequent verification dispatches required hand-edits because the description phrasing did not surface to text scanning.

**Fix design**: Extend `recordDispatch` in `spine-state.mjs`:
1. When extracted name resolves to a governance agent, append to `dispatchChain[currentStage]` (existing behavior).
2. NEW: When extracted name is null but `toolInput.subagent_type` matches a `workerTaskPackets[].ownerAgent`, look up the matching workerTaskPacket and append its `ownerAgent`. If owner is a governance agent, target the main `dispatchChain[stage]` array; otherwise record in `dispatchChain[stage]_supplementary[]`.
3. NEW: When neither resolves, record raw dispatch identifier into `dispatchChain[currentStage]_supplementary[]` so audit trails remain complete.

This addresses audit completeness without changing capability-first enforcement. The mainline gate continues to check the governance agent membership in `dispatchChain[stage]`.

**Acceptance criteria additions**:
| AC-8 | Dispatch with description containing governance agent name | dispatchChain[stage] includes that name |
| AC-9 | Dispatch with subagent_type=general-purpose AND prompt cites a workerTaskPacket id | dispatchChain[stage]_supplementary records the dispatch with the workerTaskPacket roleInstanceId |
| AC-10 | Dispatch with no recognizable owner | dispatchChain[stage]_supplementary records raw description |

### 4.4 Verifier evidence ledger contract

`config/contracts/workflow-contract.json::workerExecutionEvidenceField._meta` gains:

```json
{
  "readOnlyVerifierBypassed": "boolean (true when command was allowed via read_only_verifier slot rather than full dispatch chain)"
}
```

Reviewers must record this flag truthfully so audit trails distinguish "actual evidence" from "main-thread relayed evidence".

## 7. Open questions

1. **node -e write detection**: `node -e` is text input. Detecting write intent (e.g., `fs.writeFileSync`) inside the script string requires either regex sniffing (brittle) or treating `node -e` as deny-by-default (safe but breaks evidence patterns like `node -e 'console.log(JSON.parse(...))'`). Proposal: allow `node -e` only when command does NOT match `/fs\.writeFile|fs\.mkdir|fs\.rmdir|fs\.unlink|child_process/`.
2. **Chained npm scripts**: The `meta:check` script invokes `meta:check:runtimes && meta:check:sync-coverage && meta:validate`. These nested scripts may invoke mutating commands. Audit before whitelisting.
3. **git commands shadow risk**: `git log -p` and `git show <sha>` can dump arbitrary file content. Acceptable for review/verification but consider whether `git show <sha>:path/to/secret` needs separate handling.

## 8. Migration

- v2.3.1 ships the new gate with `readOnlyVerifierEnabled: true` only on `review` + `meta_review` + `verification` stages.
- `execution` stage stays unchanged (full dispatch chain governance).
- Backwards-compatible: old spine state without the new fields falls through to default-deny.
